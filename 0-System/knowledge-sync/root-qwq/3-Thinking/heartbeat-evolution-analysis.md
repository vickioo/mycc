## 📊 三种机制对比

### 1. mycc Scheduler - 定时任务调度器

**本质**: 基于时间表的被动触发器

```
┌─────────────────────────────────────────┐
│  tasks.md (任务配置表)                    │
│  ┌─────────────────────────────────┐    │
│  │ 时间   │ 任务    │ Skill │ 说明  │    │
│  │ 09:00 │ 晨读   │ read  │ 每日  │    │
│  │ 21:00 │ 睡前   │ sleep │ 每日  │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  scheduler.ts (轮询检查)                  │
│  - 每 60 秒检查当前时间                    │
│  - 匹配任务则执行                         │
│  - TaskLock 防止重复                     │
└─────────────────────────────────────────┘
```

**特点**:
- ✅ 配置简单 (Markdown 表格)
- ✅ 支持活跃时段过滤
- ✅ 防止重复执行锁
- ❌ 被动触发 (时间到了才执行)
- ❌ 无状态感知 (不管系统状态)
- ❌ 无进化能力 (配置不变逻辑不变)

**适用场景**: 固定时间的日常任务

---

### 2. 龙虾 (Lobster) 心跳保活 - 连接维持机制

**本质**: 基于 WebSocket 的实时连接保活

```
┌─────────────────────────────────────────┐
│  Cherry Studio / OpenClaw               │
│  ┌─────────────────────────────────┐    │
│  │  龙虾语音服务                     │    │
│  │  WebSocket 心跳 → 每 30s ping    │    │
│  │  断线重连 → 指数退避             │    │
│  │  状态同步 → 实时推送             │    │
│  └─────────────────────────────────┘    │
└─────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────┐
│  语音交互层                              │
│  - 语音输入 → STT → 文本                │
│  - 文本 → TTS → 语音输出                │
│  - 音色克隆 → 个性化语音                │
└─────────────────────────────────────────┘
```

**特点**:
- ✅ 实时连接 (WebSocket)
- ✅ 断线检测 (心跳超时)
- ✅ 自动重连 (指数退避)
- ✅ 状态同步 (实时推送)
- ❌ 单向保活 (仅维持连接)
- ❌ 无任务推进 (连接≠进展)

**适用场景**: 语音交互、实时通信

---

### 3. Mobile-Web 心跳监控 - 三方协作推进

**本质**: 状态感知 + 任务同步 + 持续推动

```
┌─────────────────────────────────────────────────────────┐
│  心跳监控 (每 5 分钟)                                      │
│  ┌───────────────────────────────────────────────────┐  │
│  │ 1. 状态检查                                        │  │
│  │    - mycc 在线？                                   │  │
│  │    - CC-Server 在线？                              │  │
│  │    - mobile-web 服务正常？                         │  │
│  │                                                   │  │
│  │ 2. 任务同步                                        │  │
│  │    - 读取 todo.md                                  │  │
│  │    - 同步到 mycc/shared-tasks/                     │  │
│  │    - 更新 COLLABORATION.md                         │  │
│  │                                                   │  │
│  │ 3. 日志记录                                        │  │
│  │    - 心跳日志 → /tmp/mobile-web-heartbeat.log      │  │
│  │    - 追加到协作文档                                │  │
│  └───────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
              ↓
┌─────────────────────────────────────────────────────────┐
│  三方协作                                                │
│  qwq ←→ mycc ←→ CC-Server                               │
│  (协调)  (开发)   (部署)                                 │
└─────────────────────────────────────────────────────────┘
```

**特点**:
- ✅ 状态感知 (检查各组件健康)
- ✅ 任务同步 (自动复制进度)
- ✅ 日志追踪 (可追溯历史)
- ✅ 安全审计 (Pre-commit Hook)
- ⚠️ 被动推进 (检查≠执行)
- ⚠️ 无自学习 (规则固定)

**适用场景**: 多 Agent 协作、分布式任务

| **日志追踪** | ⚠️ 简单 | ❌ 无 | ✅ 详细 |
| **自学习** | ❌ 无 | ❌ 无 | ❌ 无 |
| **进化能力** | ❌ 无 | ❌ 无 | ❌ 无 |

## 🤔 关键问题：如何确保持续进化？

### 当前三种机制的共同缺陷

**都没有自学习和进化能力！**

- mycc Scheduler: 配置不变，逻辑永远不变
- 龙虾心跳: 仅维持连接，不积累知识
- Mobile-Web 心跳: 检查状态，但不优化策略

## 💡 持续进化心跳机制设计

### 核心思路

```
┌─────────────────────────────────────────────────────────┐
│  进化型心跳 (Evolutionary Heartbeat)                     │
│                                                         │
│  ┌──────────────┐     ┌──────────────┐                 │
│  │  检查状态    │────▶│  执行任务    │                 │
│  └──────────────┘     └──────────────┘                 │
│         │                    │                          │
│         ▼                    ▼                          │
│  ┌──────────────┐     ┌──────────────┐                 │
│  │  记录日志    │────▶│  反思学习    │                 │
│  └──────────────┘     └──────────────┘                 │
│                          │                              │
│                          ▼                              │
│                   ┌──────────────┐                      │
│                   │  更新策略    │                      │
│                   │  (todo.md)   │                      │
│                   └──────────────┘                      │
└─────────────────────────────────────────────────────────┘
```

### 进化循环

```
检查 → 执行 → 记录 → 反思 → 更新 → (循环)
  │                                      │
  └──────────────────────────────────────┘
```

### 关键能力

1. **状态感知**: 知道当前进展到哪里
2. **反思学习**: 分析成功/失败原因
3. **策略更新**: 根据反思调整下一步
4. **知识积累**: 沉淀经验到知识库

## 🚀 实现方案

### 方案 1: 心跳 + 反思 (推荐)

每 N 次心跳后，触发一次反思：

```bash
# 每 5 次心跳 (25 分钟) 触发反思
if [ $((HEARTBEAT_COUNT % 5)) -eq 0 ]; then
    # 调用 AI 进行反思
    echo "请分析过去 25 分钟的进展：
    1. 完成了什么？
    2. 遇到了什么阻碍？
    3. 下一步应该如何调整？" | qwen -p
    
    # 更新任务列表
    update_todo_based_on_reflection
fi
```

### 方案 2: 心跳 + 知识库同步

每次心跳后，同步进展到知识库：

```bash
# 心跳检查后
if task_completed; then
    # 更新知识库
    echo "- $(date): 完成 xxx" >> /root/air/qwq/6-Diaries/2026-03/2026-03-09.md
    
    # 触发 AI 学习
    echo "今天学到了什么？" | qwen -p >> /root/air/qwq/3-Thinking/today-learning.md
fi
```

### 方案 3: 心跳 + 元认知

心跳监控自身，实现元认知：

```bash
# 心跳日志包含自身状态
log "心跳周期：${INTERVAL}s"
log "连续运行：${UPTIME}"
log "任务推进：${TASKS_COMPLETED}/${TASKS_TOTAL}"
log "反思触发：${LAST_REFLECTION}"

# 定期分析心跳效果
if [ $((UPTIME % 3600)) -eq 0 ]; then
    echo "心跳机制是否有效？如何优化？" | qwen -p
fi
```

## 🎯 结论

**当前 Mobile-Web 心跳机制**相比 mycc Scheduler 和龙虾心跳：

| 对比项 | 优势 | 劣势 |
|--------|------|------|
| vs Scheduler | ✅ 状态感知<br>✅ 安全审计<br>✅ 日志追踪 | ❌ 无定时任务执行 |
| vs 龙虾心跳 | ✅ 任务推进<br>✅ 三方协作<br>✅ 可追溯 | ❌ 无实时连接 |

**但三者都缺乏持续进化能力**。

**建议**: 在 Mobile-Web 心跳基础上，增加**反思学习循环**，实现：
- 每 N 次心跳触发反思
- 反思结果更新任务策略
- 知识沉淀到知识库
- 元认知监控心跳效果

这样才能确保**持续进化的思考能力**。

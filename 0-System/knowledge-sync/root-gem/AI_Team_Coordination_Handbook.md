# AI Hub & MyCC 联合架构与跨端协同工程手册

**文档版本**: V1.0  
**更新日期**: 2026-03-20  
**核心目标**: 规范化多端（U22/Victory/Surface/CC）代码资产管理，明确 AI 智能体群分工，建立高效、边界清晰的 API 调度与算力分配体系。

---

## 一、 架构哲学与仓库管理规范

必须严格厘清两个核心代码库的设计哲学与目录体系，严禁互相污染：

### 1. `aihub` (家庭/极简中枢系统)
*   **设计哲学**: 原生极简架构，Monorepo 体系。追求高内聚、低耦合，主要用于家庭内网中枢展示、轻量级 Web 服务和状态看板。
*   **目录体系**: 扁平化，不要生搬硬套复杂的分类体系（不需要 `0-System`, `2-Projects` 这种前缀）。
*   **存储位置**: 核心源码开源至 GitHub 排榜；涉及真实 IP、密钥等高敏数据的“完整版/私有版”统一存放在局域网高可用家庭服务器（Victory / Surface）的私有 Lab 仓库中跨设备同步。

### 2. `mycc` (深度技能与代理执行层)
*   **设计哲学**: 深度系统化、高度模块化的复杂 Agent 技能架构。主要为 Claude Code 等强力 AI 提供操作手脚。
*   **目录体系**: 严格遵循体系化分类（如 `0-System`, `4-Assets`, `.claude/skills` 等）。
*   **代码源流管理**:
    *   **主库指向**: 将原先指向第三方开源库 (`Aster110/mycc`) 的远程 origin 改为用户自身名下的 Fork 仓库 (`vickioo/mycc`)。
    *   **上游同步**: 将第三方源配置为 `upstream`，这样既可以在自己的 Fork 中安全定制专属技能，又能第一时间 `git fetch upstream` 获取官方的底层版本更新和新特性。

## 二、 智能体群 (AI Agents) 角色分配与算力调度

合理利用云端强算力与家庭边缘端设备，建立边界清晰的 AI 协同作战总线。

### 1. 角色与节点分工

| 节点 (设备) | 角色定位 | 算力特征 | 核心承担与分工 |
| :--- | :--- | :--- | :--- |
| **CC (云端服务器)** | **重装主将 (Logic & Skills)** | 强算力 / 宽带大 | 部署 `mycc`，运行复杂逻辑技能 (如 PDF/PPTX 解析)。作为飞书-钉钉双向机器人 Webhook 的处理中枢。API 代理网关枢纽。 |
| **U22 (本地/Termux)** | **现场指挥官 (Exec & Orchestration)** | 中边缘算力 / 高机动 | Gemini CLI 常驻地。负责系统级排障、底层网络打通、脚本注入、跨库梳理的现场执行。作为开发与指令下发的第一入口。 |
| **Victory (Windows)** | **数据大本营 (Data & Archiving)** | 24/7 稳定在线 / 大存储 | 作为本地私有 Lab 仓库的 Git 中心。存放全量高敏数据和配置。承载 Windows 原生 AI 能力整合。 |
| **Surface (Ubuntu)** | **低功耗核心备用站 (Edge AI)** | 极低电费 / 备用网关 | 离线与边缘 AI 核心站。运行本地 Qwen 模型、Lobster 龙虾语音服务。在云端 API 抖动时提供稳定托底能力。 |

### 2. API 供应与边界管理方案

*   **公司/云端边界**: CC 服务器作为公网与 API 供应商的桥梁，负责对接 Anthropic (Claude)、OpenAI 等高昂算力 API，并做好用量统计 (`0-System/qwen-real-api.md` 等记录)。
*   **家庭/内网边界**: 敏感数据处理、局域网内设备巡检、以及日常琐碎的高频轻量级对话，优先通过 Surface (Qwen) 和内部 API 网关处理，保护隐私并降低 Token 损耗。
*   **API 统一调度**: 依托 OneAPI 等网关工具，在 CC 或 Victory 上建立统一的 API 路由表，按需 fallback（如高阶推理分配给 Claude 3.5 Sonnet，日常补全 fallback 至本地 Qwen 或廉价 API）。

2. 将边缘侧高频任务（如系统巡检）固化为 Cron 定时任务，由 U22 或 Surface 自动执行并汇报至协同总线。
